# Architecture

This Next.js 16 application is designed with a feature-based architecture to prioritize scalability, modularity, and AI-assisted development.

## Folder Structure

- **`/src/app`**: Contains Next.js App Router specific files (pages, layouts, API routes). We use nested layouts for consistent UI shells.
- **`/src/components/ui`**: Base primitive components generated by `shadcn/ui`. These are the building blocks of the design system. Do not write business logic here.
- **`/src/components/shared`**: Composed components used across multiple features (e.g., `Navbar`, `Footer`, `MDXRenderer`).
- **`/src/features`**: Feature modules containing domain-specific components, hooks, and utilities. For example, `/src/features/case-studies` handles everything related to displaying and protecting case studies.
- **`/src/content`**: MDX content files and schemas.
- **`/src/config`**: Global configuration files (e.g., site metadata, navigation links).
- **`/src/hooks`**: Generic, reusable React hooks not tied to a specific feature.
- **`/src/lib`**: Utility functions and third-party wrappers.
- **`/src/types`**: Global TypeScript types and interfaces.

## Key Architectural Principles

1. **Server-First Approach**: We leverage React Server Components (RSC) to minimize client-side JavaScript. Only components that require interactivity or browser APIs should be marked with `"use client"`.
2. **Separation of Concerns**: UI components should be isolated from data fetching. Data fetching happens in Server Components or dedicated custom hooks.
3. **Modularity**: Code should be grouped by feature rather than by file type (e.g., `/features/auth` instead of dumping everything into a massive `/components` folder).
4. **Security**: Password-protected case studies use Next.js Middleware and secure HttpOnly cookies to ensure content is never exposed to unauthorized clients.
